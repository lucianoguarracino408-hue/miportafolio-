01 — Contexto y problema

¿Qué problema
resuelve?

Fotografiar páginas de libros, documentos y recibos con el celular produce imágenes inconsistentes. La calidad depende de la luz, el pulso y el ángulo. CELI AI automatiza la mejora.

💡

Iluminación variable

Las fotos tomadas en ambientes con luz artificial o natural desigual presentan zonas sobreexpuestas y subexpuestas en la misma imagen.

🌀

Ruido y desenfoque

El movimiento de la mano y la baja apertura de las cámaras móviles generan imágenes borrosas y con ruido digital, especialmente con poca luz.

⚙️

Parámetros manuales

Las apps existentes obligan al usuario a ajustar manualmente el contraste, brillo y nitidez. No existe un sistema que detecte el contexto y calibre solo.

La solución no es solo procesar la imagen. Es entenderla primero, y luego aplicar el tratamiento exacto que necesita según lo que muestra.

CELI AI resuelve esto con un sistema de tres capas: una app móvil que captura automáticamente cuando detecta foco, un módulo de inteligencia visual (Vini) que analiza la imagen, y un servidor con motores OpenCV especializados por tipo de contenido.

03 — Módulo Vini

Vini: inteligencia
visual

Visual Intelligence Node — el cerebro del sistema. Analiza cada imagen antes de procesarla y decide automáticamente qué parámetros aplicar. Sin intervención del usuario.

BlurDetector

Mide el desenfoque usando la varianza del Laplaciano. Un Laplaciano detecta bordes; si la varianza es baja, la imagen tiene pocos bordes definidos → está borrosa.

Clasifica en 4 niveles y sugiere el factor de sharpening correspondiente.

muy_borrosa → sharpness 1.8
nítida → sharpness 1.0
BrightnessDetector

Analiza la distribución tonal completa: brillo medio, desviación estándar, ratio de píxeles oscuros y sobreexpuestos.

Calcula un exposure score con función gaussiana centrada en 128 (brillo ideal).

score 0 (malo) → 1 (ideal)
NoiseDetector

Combina tres métodos para mayor precisión: diferencia con Gaussiano (50%), energía del Laplaciano (30%) y varianza local en bloques de 16×16 (20%).

El score compuesto determina la fuerza de denoise recomendada.

score 0–100 → denoise 0–0.95
FaceDetector

Usa el clasificador Haar Cascade de OpenCV, incluido en la librería sin dependencias extra. Detecta caras frontales y mide su área relativa en la imagen.

Si la cara más grande ocupa más del 15%, activa modo retrato.

portrait / group / no_faces

04 — Procesamiento OpenCV

Pipeline de
3 motores

Una vez que Vini determina el contexto, la imagen pasa por tres motores OpenCV encadenados. Cada uno tiene una responsabilidad específica y opera sobre el resultado del anterior.

Análisis Vini detección

Vini analiza las 5 métricas clave (brillo, ruido, contraste, bordes, saturación) y determina el modo óptimo. A partir de este punto, todos los parámetros están calibrados para este tipo específico de imagen.

Motor 1 — Denoise ruido

Reducción de ruido selectiva usando fastNlMeansDenoisingColored. La fuerza (parámetro h) se escala entre 4 y 12 según el noise_score de Vini. No aplica si el ruido es menor a 5%, preservando detalles finos en imágenes limpias.

Motor 2 — Rango dinámico tono

Cuatro sub-pasos: (a) corrección de viñeta gaussiana para compensar la caída de luz en bordes de lente móvil, (b) deskew suave para corregir inclinaciones menores a 15°, (c) tone mapping zonal independiente para sombras/medios/luces en canal L del espacio LAB, (d) CLAHE adaptativo con clip distinto para centro y bordes de la imagen.

Motor 3 — Nitidez por modo modo

El pipeline de nitidez varía según el contexto detectado. book: Unsharp Mask refinado sobre canal L + White Balance Grey World. photo: sharpening edge-aware en espacio LAB + boost de saturación + corrección gamma. document: sharpening agresivo con máscara de bordes Sobel + binarización adaptativa para máximo contraste texto/fondo.

Codificación y entrega salida

La imagen mejorada se codifica a JPEG con calidad optimizada (85–95 según modo), se genera un thumbnail 200×200, se calcula el hash SHA-256 para la URL firmada con HMAC, y se almacena en caché LRU. La respuesta JSON incluye la imagen en base64, el thumbnail, las métricas de mejora antes/después, y la URL firmada de acceso directo.

Todo el sistema corre en contenedores Docker orquestados con Compose. Un solo comando levanta todos los servicios con sus dependencias correctas.

# Verificar que funciona:

Los servicios web y worker comparten un volumen Docker (shared_data) para que ambos accedan a las mismas imágenes procesadas. El health check de PostgreSQL usa pg_isready para garantizar que el servidor no arranque antes de que la base esté disponible.

06 — Reflexión técnica

Qué aprendí
construyendo esto

Cada componente del proyecto presentó un desafío técnico específico. Estos son los más importantes, con la solución que encontré.

⚡
Concurrencia en un servidor de imágenes OpenCV es completamente síncrono. Si el pipeline de una imagen tarda 2 segundos, bloquea todo el event loop de FastAPI para las demás solicitudes. La solución fue usar asyncio.to_thread() para ejecutar el pipeline en un thread pool, más un Semaphore de 4 para no saturar la CPU con demasiados procesos paralelos.
📱
Procesamiento en tiempo real en Flutter Medir la nitidez de cada frame de cámara en el hilo principal causaba drops de FPS. La solución fue un Isolate persistente que recibe el plano Y del YUV420 con TransferableTypedData (zero-copy), calcula la varianza del Laplaciano, y devuelve solo un número. El hilo principal nunca se bloquea.
🔬
Diseño de detectores modulares En lugar de un único clasificador monolítico, separé la lógica en detectores independientes (blur, brillo, ruido, caras) que cada uno devuelve un dict estructurado. El ContextClassifier los fusiona después. Esto permite testear cada detector por separado y mejorar uno sin afectar los demás.
🔐
Seguridad en URLs de imágenes Las imágenes procesadas son privadas. Para servir acceso temporal usé URLs firmadas con HMAC-SHA256: el servidor genera un token que incluye el hash de la imagen y un timestamp de expiración. Si alguien modifica la URL, la firma no coincide y el acceso es denegado.
🎨
Por qué el espacio de color LAB para nitidez Aplicar sharpening en BGR crea artefactos de color en los bordes (halos de color). Al trabajar en el espacio LAB y aplicar los cambios solo sobre el canal L (luminancia), la saturación y el tono original se preservan. El ojo humano percibe la nitidez principalmente en luminancia, no en color.
📦
Escritura atómica para evitar archivos corruptos Si el servidor recibe un kill signal mientras escribe una imagen, el archivo queda truncado y corrupto. La solución es escribir primero en un archivo temporal (NamedTemporaryFile) y luego moverlo con shutil.move(). El movimiento es atómico en el mismo filesystem, así el archivo aparece completo o no aparece.

07 — Cierre

Conclusión
del proyecto

Un sistema real, no un prototipo

CELI AI no es un experimento académico aislado. Es un sistema de producción con autenticación, manejo de errores, límites de concurrencia, caché, logging, health checks y deploy en Docker. Cada decisión técnica tuvo una razón: el semáforo para no saturar la CPU, el Isolate para no bloquear la UI, el espacio LAB para no arruinar los colores, la escritura atómica para no corromper archivos.

El módulo Vini fue el aprendizaje más importante: construir un sistema de análisis que funciona en capas — detectores independientes que se combinan en un clasificador — es una forma de pensar que se aplica a cualquier problema de IA, no solo a imágenes.

El siguiente paso es agregar un modelo de deep learning ligero (MobileNetV3) para reemplazar las reglas manuales del ContextClassifier y mejorar la detección de contextos complejos como low_light o product_photo.

Todos los archivos del proyecto están documentados y el sistema corre con un único docker-compose up. El servidor expone documentación automática de la API en /docs (Swagger UI generado por FastAPI).
